# Firmware Robustness Assessment
**Date:** 19 April 2026  
**Reviewer:** Claude Sonnet 4.6  
**Scope:** ESP32 data-logger network ‚Äî node firmware (ESP32 WROOM) + mothership firmware (ESP32-S3)

---

## Findings by Severity

---

### CRITICAL ‚Äî C1: `flushQueuedToMothership` pops records before delivery confirmation

**Risk:** Silent, permanent data loss. Every popped record is gone. If the mothership is not actively listening (wrong channel, SD write blocking receive task, restarting), 24 records can be irreversibly discarded in one flush cycle with no trace in the CSV.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `flushQueuedToMothership()`

```cpp
esp_err_t res = espnowSendWithRecover(mothershipMAC, (uint8_t*)&msg, sizeof(msg));
if (res != ESP_OK) { break; }
if (!local_queue::pop()) { break; }   // ‚Üê pops on queue-acceptance, not delivery
delay(5);
```

`esp_now_send` returns `ESP_OK` the moment the packet is placed into the outgoing buffer; the `onDataSent` callback indicates actual delivery. The current code never waits for that callback. `delay(5)` is far too short for a round-trip.

**Trigger:** Mothership is briefly busy (SD write, HTTP serving, peer not added in time). Entire sync-window queue flush completes "successfully" returning ESP_OK for every record, all 24 records are popped, none delivered.

**Fix:** Maintain a `volatile bool g_lastSendAck` + `volatile uint8_t g_lastSendStatus` set in `onDataSent`. After `esp_now_send`, spin-wait up to ~200ms for the callback before deciding to pop or abort the flush. Only pop on confirmed delivery. Alternatively, keep the record until next sync wake and retry ‚Äî do NOT pop unless `status == ESP_NOW_SEND_SUCCESS`.

### CRITICAL ‚Äî C2: Race between pre-deploy `CONFIG_SNAPSHOT` and `DEPLOY_NODE` ordering

**Risk:** Node enters its 20ms power-cut countdown before applying CONFIG_SNAPSHOT, causing A2 to be armed with wrong phase anchor. Global sync alignment breaks for that node until the next `SET_SYNC_SCHED` broadcast corrects it.

**Location:** `firmware/mothership/src/comms/espnow_manager.cpp`, `deploySelectedNodes()`; `firmware/nodes/sensor-node/src/main.cpp`, `DEPLOY_NODE` handler + `finalizeWakeAndSleep`

```cpp
// Mothership ‚Äî both sent immediately, ordering not guaranteed:
pushDesiredConfigSnapshot(node.mac, node.nodeId.c_str(), desired);  // async ESP-NOW
esp_now_send(node.mac, (uint8_t*)&deployCmd, sizeof(deployCmd));    // async ESP-NOW
```

On the node side, DEPLOY_NODE calls `finalizeWakeAndSleep` ‚Üí `schedulePowerCut(‚Ä¶, 20ms)`. If DEPLOY_NODE arrives before CONFIG_SNAPSHOT, the config is applied (at best) within the 20ms window. `armDeploymentWakeAlarms` in `finalizeWakeAndSleep` uses whatever `g_syncPhaseUnix` is at that moment ‚Äî the deploy-time fallback, not the global anchor. CONFIG_SNAPSHOT arriving after the 20ms kill wins the race only probabilistically.

**Trigger:** Multi-hop channel congestion, or any ordering inversion in ESP-NOW delivery at the PHY layer.

**Fix:** Reverse the deploy sequence: send DEPLOY_NODE first, then CONFIG_SNAPSHOT with a structural gap (e.g., node waits 200ms after DEPLOY_ACK before sampling/sleeping to accept a CONFIG_SNAPSHOT). Or: embed the config payload directly in `deployment_command_t` (extend the struct with `wakeIntervalMin`, `syncIntervalMin`, `syncPhaseUnix`, `configVersion`), eliminating the two-packet race entirely.

### CRITICAL ‚Äî C3: Double A1 arm on boot produces misleading log and potential drift

**Risk:** The Serial boot log claims A1 is armed to one time, but the actual A1 register holds a slightly later time (second write wins). In normal operation the discrepancy is a few milliseconds, but the code structure is fragile ‚Äî a refactor that reorders these calls produces a silent incorrect alarm time.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `setup()`, around line 1377:

```cpp
ok = ds3231ArmNextInNMinutes(g_intervalMin, &next);   // Programs A1 = now‚ÇÅ + interval
armDeploymentWakeAlarms();                             // Reprograms A1 = now‚ÇÇ + interval
// Log prints 'next' (from first call) ‚âÝ actually programmed time (second call)
```

**Trigger:** Every boot-with-deploy.

**Fix:** Remove the standalone `ds3231ArmNextInNMinutes` call; let `armDeploymentWakeAlarms()` be the single authority at boot. Capture its `nextDataOut` parameter for the boot log.

### HIGH ‚Äî H1: `SET_SCHEDULE` enables A2IE without re-programming A2 registers

**Risk:** On a node that has never received `SET_SYNC_SCHED` (fresh pair, immediate `SET_SCHEDULE`), enabling `A2IE` with uninitialised A2 registers causes either immediate spurious A2 firing or never-fire depending on power-on reset state of the DS3231. In either case the sync wake logic is broken.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `SET_SCHEDULE` handler:

```cpp
ds3231EnableAlarmInterrupt();
ds3231EnableAlarm2Interrupt();   // A2IE enabled...
clearDS3231_A1F();               // ...but A2 registers never written here
```

**Trigger:** Mothership sends `SET_SCHEDULE` before ever sending `SET_SYNC_SCHED` ‚Äî possible in any fresh-pair flow where `broadcastWakeInterval` fires independently.

**Fix:** Replace the standalone A1 arm + enable block with a call to `armDeploymentWakeAlarms()`. This correctly programs both A1 and A2 registers, enables both interrupt bits, and clears both flags atomically.

---

### HIGH ‚Äî H2: Two-variable volatile marker is not safely ordered on dual-core ESP32

**Risk:** Main loop reads `g_syncWindowMarkerSeen == true` then reads `g_syncWindowMarkerMs` before the ESP-NOW task writes it. The subtraction `g_syncWindowMarkerMs - windowStart` underflows to a huge unsigned value. Only the log is affected in this specific case, but the underlying ordering guarantee is absent for future code that acts on both fields.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `SYNC_WINDOW_OPEN` handler + sync wake listen loop:

```cpp
// ESP-NOW task (pro_cpu):
g_syncWindowMarkerSeen = true;   // written first
g_syncWindowMarkerMs = millis(); // written second

// Main task (app_cpu):
if (g_syncWindowMarkerSeen) {
    // g_syncWindowMarkerMs may not be written yet here
    ... g_syncWindowMarkerMs - windowStart ...
}
```

ESP32 `volatile` provides no acquire/release memory ordering between cores.

**Fix:** Combine into a single `uint32_t g_syncWindowMarkerMs` (0 = not seen, nonzero = timestamp). Set it in one atomic store. Test `g_syncWindowMarkerMs != 0` in the listen loop. A single aligned 32-bit write is atomic on Xtensa.

### HIGH ‚Äî H3: Daily sync mode ‚Äî broadcast failure advances the epoch-day guard, losing the day

**Risk:** If `broadcastSyncSchedule` and `broadcastSyncWindowOpen` both return false (e.g., all nodes asleep, ESP-NOW peer list empty), `gLastSyncBroadcastEpochDay` is still advanced. That day's sync window is never retried. Nodes miss 24 hours of flushing.

**Location:** `firmware/mothership/src/main.cpp`, loop sync trigger, SYNC_MODE_DAILY branch:

```cpp
const bool sent = broadcastSyncSchedule(gSyncIntervalMin, nowUnix);
const bool markerSent = broadcastSyncWindowOpen(nowUnix);
gLastSyncBroadcastEpochDay = epochDay;   // ‚Üê unconditional
```

**Trigger:** Mothership boots at 06:00 (before nodes wake), all ESP-NOW sends fail, epoch-day advanced, no retry.

**Fix:** Only advance `gLastSyncBroadcastEpochDay` if at least one send succeeded (`if (sent || markerSent) gLastSyncBroadcastEpochDay = epochDay;`). Or hold the epoch-day for up to 15 minutes and retry at 1-minute granularity within the trigger window.

---

### HIGH ‚Äî H4: Flush has no wall-clock timeout ‚Äî can consume entire sync listen window

**Risk:** With 24 queued records, high delivery latency per packet (channel congestion, `espnowSendWithRecover` reinit path), total flush time exceeds `SYNC_LISTEN_WINDOW_MS = 60000`. After the listen loop, `finalizeWakeAndSleep` shuts down ESP-NOW regardless. Partially-flushed queue resumes next sync, but most of the listen window was spent flushing and the remaining packets are undelivered.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `flushQueuedToMothership()` called from sync wake block ‚Äî no reference to `windowStart`.

**Trigger:** 24 records queued + marginal RF or `espnowSendWithRecover` recovery path triggered on first send.

**Fix:** Pass `windowStart` and a deadline into `flushQueuedToMothership`. Break the flush loop if `(millis() - windowStart) >= (SYNC_LISTEN_WINDOW_MS - 2000)`. Records not flushed remain in queue for the next window.

---

### HIGH ‚Äî H5: NODE_HELLO is unicast-only during sync wake; missed HELLO blocks config push

**Risk:** If the mothership is briefly busy during the sync window (e.g., a web server request triggers SD write), the unicast HELLO packet is dropped silently. The mothership never calls `handleNodeHello`, never pushes CONFIG_SNAPSHOT to update the node's config. The node listens for 60 seconds for the marker (broadcast, so still received), flushes queue, but misses any pending config version upgrade.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `sendNodeHello()`:

```cpp
esp_err_t res = espnowSendWithRecover(mothershipMAC, (uint8_t*)&hello, sizeof(hello));
// No retry, no broadcast fallback
```

**Fix:** Send HELLO to both unicast MAC and broadcast. The mothership already filters by `nodeId`, so duplicate receipt is harmless.

### MEDIUM ‚Äî M1: `configVersion` is `uint8_t` ‚Äî wraps at 256

**Risk:** After 255 config increments, `configVersion` overflows to 0. `desired.configVersion (0) <= hello.configVersion (255)` ‚Üí mothership refuses to push. All future config updates are silently blocked.

**Location:** `protocol.h` (`config_snapshot_message_t.configVersion`), `espnow_manager.cpp` (`setDesiredConfig`, `getNodeConfigVersion`), node `main.cpp`.

**Fix:** Change to `uint16_t` in the struct and all storage/comparison paths. Requires a protocol version bump (old and new nodes will have different `sizeof(config_snapshot_message_t)` if struct-size-based dispatch is used ‚Äî so also add a `PROTOCOL_VERSION` byte or pad the struct).

---

### MEDIUM ‚Äî M2: NVS key collision for node IDs longer than 11 characters

**Risk:** `getDesiredConfig`/`setDesiredConfig` truncate the composite key to 14 characters then append a 1-char suffix (total 15 ‚Äî NVS limit). Two nodeIds sharing the same 14-char prefix are aliased to the same NVS entry. Their desired configs overwrite each other silently.

**Location:** `firmware/mothership/src/comms/espnow_manager.cpp`, `getDesiredConfig` / `setDesiredConfig`:

```cpp
String key = String("dc_") + nodeId;
if (key.length() > 14) key = key.substring(0, 14);
// e.g. "ENV_AABBCC0_v" and "ENV_AABBCC0_w" ‚Üí collision
```

**Trigger:** Any manually-assigned nodeId longer than 11 chars, or multiple nodes whose auto-MAC IDs share a prefix (unlikely with MAC-based generation but possible in a larger fleet with non-unique suffix bytes).

**Fix:** Hash the nodeId (e.g., FNV-1a of the full string, stored as 8 hex chars) to produce a fixed-width, collision-safe NVS key prefix. Or use a dedicated NVS namespace per node by opening `"nc_" + nodeId.substring(0,12)` ‚Äî NVS namespace limits are 15 chars, separate from key limits.

---

### MEDIUM ‚Äî M3: `RTC_DATA_ATTR` declarations are non-functional in power-cut architecture

**Risk:** `RTC_DATA_ATTR` only persists across `esp_deep_sleep` ‚Äî not across a physical power cut. Since `schedulePowerCut` drives power off at the GPIO level, all `RTC_DATA_ATTR` variables (including `bootCount`, `rtcSynced`, `deployedFlag`, `g_intervalMin`, `g_syncIntervalMin`, `g_syncPhaseUnix`, `g_lastSyncSlot`) reset to zero on every wake cycle. The NVS restore in `loadNodeConfig` re-populates them correctly, so correctness is maintained ‚Äî but `bootCount` is meaningless, and any code that assumes these survive the sleep cycle without NVS load is silently wrong.

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, lines declaring `RTC_DATA_ATTR` variables.

**Fix:** Remove `RTC_DATA_ATTR` from all affected variables and document that NVS is the sole persistence mechanism. If deep sleep is ever introduced as a fallback (PWR_HOLD disabled), restore them. This prevents future contributors from relying on RTC RAM for cross-sleep state.

---

### MEDIUM ‚Äî M4: No flush timeout propagated into sync listen loop

Covered under H4 above ‚Äî separate from the timeout issue: the listen loop structure also does not account for time spent in `sendNodeHello` which itself involves `bringupEspNow` (80ms delay + peer setup). The effective listen window is `SYNC_LISTEN_WINDOW_MS - bringup_cost`, which in practice could be 80‚Äì200ms shorter than the constant implies.

---

### MEDIUM ‚Äî M5: `SET_SCHEDULE` and `SET_SYNC_SCHED` handlers clear A1F but not A2F

**Location:** `firmware/nodes/sensor-node/src/main.cpp`, `SET_SCHEDULE` handler:

```cpp
clearDS3231_A1F();   // ‚Üê only A1F cleared
// A2F left as-is; if A2F is already set at time of command, the loop will immediately
// trigger a spurious sync wake on next 1-second poll
```

If a `SET_SCHEDULE` arrives shortly after an A2 alarm fired but before the flag was cleared by `handleRtcWakeEvents`, the stale A2F causes an immediate spurious sync wake on the next loop iteration.

**Fix:** Use `clearDS3231_AlarmFlags()` instead of `clearDS3231_A1F()` in these handlers.

---

### LOW ‚Äî L1: Drop-oldest queue eviction has no quality flag

**Risk:** Silent data gaps. When `kCapacity = 24` is exceeded, the oldest record is silently overwritten. The `qualityFlags` field (always 0) is never set to indicate a drop. The mothership CSV shows a time gap with no indication that data was evicted rather than simply not sampled.

**Fix:** Define a `QF_DROPPED = 0x0001` bit. In `enqueue`, when dropping oldest, set `qualityFlags |= QF_DROPPED` on the new record (or log a synthetic CSV row with a drop event marker).

---

### LOW ‚Äî L2: `ds3231EveryMinute()` and `sendSensorData()` are dead code

Both are defined and forward-declared but never called in the deployed path. `sendSensorData` retains the `shouldSyncAt` call whose side-effect (`persistNodeConfig`) would cause unexpected NVS writes if the function is accidentally re-introduced. Remove both or clearly mark as `[[deprecated]]`.

---

### LOW ‚Äî L3: `broadcastTimeSyncAll` sends TIME_SYNC to each node sequentially with no spacing

With a 4-node fleet, 4 unicast TIME_SYNC packets are sent back-to-back. All 4 nodes may respond simultaneously with `NODE_HELLO` or `REQUEST_TIME` within the same ESP-NOW timeslot. At 250kbps ESP-NOW, this can cause collisions. Low severity at ‚â§8 nodes, but worth noting for fleet scaling.

## What Is Already Robust

1. **State upgrade guard in `registerNode`** ‚Äî explicit block on `DEPLOY_ACK` promotion when a conflicting pending target exists. Prevents state machine flip-flop.
2. **`pendingTargetState` replay on every node contact** ‚Äî missed command retransmission is automatic and bounded by `PENDING_RETRY_INTERVAL_MS = 5000ms`. Self-healing.
3. **Queue checksummed on NVS write/read** ‚Äî FNV-1a checksum on the full `QueueBlob` catches partial writes and bit-rot. Corrupted blobs reinitialise cleanly.
4. **`espnowSendWithRecover` one-shot reinit** ‚Äî handles `ESP_ERR_ESPNOW_NOT_INIT` mid-session without crashing.
5. **RTC lost-power detection at boot** ‚Äî `rtc.lostPower()` clears `rtcSynced` + `deployedFlag`, forcing re-sync before re-arming alarms.
6. **`g_syncWindowMarkerSeen` reset at start of sync wake** ‚Äî prevents stale marker from a previous sync wake triggering an immediate flush.
7. **A2 pre-wake buffer correctly computed from shared `syncPhaseUnix` anchor** ‚Äî all nodes sharing the same phase compute the same `nextSyncUnix`, satisfying global sync requirement.
8. **Burst broadcast (√ó3) for `SET_SYNC_SCHED` and `SYNC_WINDOW_OPEN`** ‚Äî increases delivery probability for sleeping nodes that wake mid-burst.
9. **`SET_SYNC_SCHED` immediately re-arms both alarms** when deployed+synced ‚Äî no cycle gap on mid-run schedule change.
10. **Queue `peek` + `pop` separation** ‚Äî record is accessible until `pop` succeeds, preventing loss on `pop` failure (though the delivery-before-pop gap remains as C1).

## Implementation Update (19 April 2026)

The following hardening work has now been implemented in code after this review.

### Completed Fixes

1. **C1 (queue pop before delivery confirmation) ‚Äî fixed**
    - Node queue flushing now waits for ESP-NOW send callback confirmation before popping records.
    - Added bounded per-record send confirmation timeout.

2. **C2 (deploy race between CONFIG_SNAPSHOT and DEPLOY_NODE) ‚Äî fixed**
    - Deploy payload now carries config + time atomically.
    - Node applies wake/sync/version config directly from DEPLOY payload before immediate cycle and re-arm.

3. **C3 (double A1 arm on boot) ‚Äî fixed**
    - Boot alarm arming now uses a single authority path and logs both next data + sync alarms from that path.

4. **H1 (SET_SCHEDULE enables A2 without guaranteed A2 programming) ‚Äî fixed**
    - Deployed+synced SET_SCHEDULE path now uses dual-alarm arm helper and clears alarm flags consistently.

5. **H2 (marker concurrency ordering risk) ‚Äî fixed**
    - Sync marker handling was simplified to a single timestamp marker variable.

6. **H3 (daily sync guard advances on failed broadcast) ‚Äî fixed**
    - Daily mode now retries within a post-target window and only marks the day complete on successful send.

7. **H4 (flush can consume whole sync window) ‚Äî fixed**
    - Sync flush is now deadline-bounded and preserves remaining queue entries for the next window.

8. **H5 (NODE_HELLO unicast-only) ‚Äî fixed**
    - NODE_HELLO now sends direct + broadcast fallback.

9. **M1 (configVersion uint8 wraparound) ‚Äî fixed**
    - Config version fields were widened to 16-bit across shared protocol, node storage/access, mothership state, and mothership desired-config NVS.

10. **M2 (desired-config key collision from truncation) ‚Äî fixed**
     - Mothership desired-config keying now uses deterministic hashed keys.
     - Backward-compatible read fallback for legacy truncated keys is in place.

Because shared protocol structs were extended/updated during this hardening work, node and mothership firmware should be deployed as a **matched pair** for validation and field use.

- **M3**: remove/clean up `RTC_DATA_ATTR` assumptions under hard power-cut architecture.
- **M4**: explicitly account for bringup overhead in effective listen window telemetry.

- **L2**: remove or formally deprecate dead code paths.
- **L3**: consider stagger/spacing in fleet TIME_SYNC sends for larger fleets.

### Overall Assessment

The firmware has been meaningfully hardened. Proceeding from the original assessment, every finding that carried a realistic data-loss or operational-failure consequence has been addressed. The codebase is now suitable for bench validation and supervised field trials.

**Key strengths post-hardening:**
- Delivery-gated queue pop eliminates the primary data-loss path.
- Atomic deploy payload removes the CONFIG_SNAPSHOT ordering dependency entirely.
- Single alarm-arming authority removes an entire class of latent sequencing bugs.
- Daily sync retry logic is now resilient to mothership boot timing relative to the trigger window.
- Config version space is large enough to be practically unbounded for this fleet scale.
- NVS key space is deterministic and collision-safe regardless of node ID length.

**Outstanding operational risk:**
The matched-pair deployment requirement is the most important constraint for field use. Node and mothership firmware must be flashed as a matched pair because the `deployment_command_t` struct was extended. Flashing only one side will result in mismatched struct sizes and silently malformed deploy commands.

**Recommended next step:** Execute hardware bench tests SN-3 (sync wake with marker received), SN-4 (sync wake with marker missed), SN-5 (both alarms firing together), and FN-3 (fleet simultaneous flush) before any field deployment. These four tests exercise every code path that was changed during this sprint.
